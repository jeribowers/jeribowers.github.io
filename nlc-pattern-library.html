NLC Pattern Library

Aligning visual and user experience patterns to achieve cohesive user-centered experiences

My role: lead, visual design, HTML / CSS, Sass

The pattern library was a new initiative to align visual and user experience patterns across four dispersed teams to achieve compelling and cohesive user-centered experiences. The secondary goal was to create a pattern library so that designers could rapidly prototype and share consistent modularized patterns with their development team.

Design audit

We started with a design audit to analyze what was currently being used in our products. By documenting each of the elements and patterns in Mural.ly, we could easily spot overlaps and inconsistencies. We used Brad Frost’s Atomic Design methodology to organize our patterns, labeling each as an atom, molecule, or organism. We also included a Watson specific category because Watson's cognitive capabilities aren’t common vernacular.

Each pattern was prioritized based on importance and usage and assigned to a specific sprint. However, the schedule would adapt as we progressed through each sprint if priorities shifted.

Using GitHub to establish a workflow

Once we had a catalog of all the common UI patterns we needing to include into the library, each item was added as a specific issue in GitHub. There were 4 main areas we wanted to align on: guidelines for usage, variations and specific use cases, acceptance criteria, and visual styling.

Adding this template to the first comment of each issue, helped to establish a baseline and facilitate consistency. Each issue would then be labeled with a product label, the type of element it was (according to Brad Frost’s Atomic Design), and assigned to a milestone. Each milestone aligned to the 3-week sprint schedule our product teams used, which also included offering managers and developers.

In the beginning of the sprint, each issue would be assigned to a visual designer.

Their responsibilities included:

  1. Verify that screenshots had been collected for that issue
  2. Take a first pass at writing usage guidelines
  3. Request approval of visual styling and guidelines from team members

The visual designer that was assigned to the issue wasn’t necessarily responsible for completing it on their own. Their main priority was to help facilitate the completion of that issue. The other team members (included UX, visual, research, and developers) were responsible for contributing relevant screenshots, questions, or comments. We found that it was best to include full page screenshots so that everyone could understand the context. Patterns shown in isolation weren't always adequately reviewed.

Since our team was dispersed across the US and the UK, we would keep the conversations contained in GitHub. This allowed us to have free flowing discussions even if team members were available at different times.

Once an issue was approved (which required sign-off by at least one visual and UX designer), a coder would be assigned. That coder would then create a feature branch, make code changes, and submit a pull request.

After I reviewed and pulled in the code change, the pattern was added to the NLC pattern library and would be followed in future designs and prototypes. If a pattern needed to be reviewed again, anyone could reopen the issue and add to the current milestone.
